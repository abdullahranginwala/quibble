# M1 — `pkg/render`: markdown → HTML, theme contract, paper theme

**Goal:** the independently importable rendering library plus the machinery `build` and `serve` will call. After M1, `render.New(...).RenderDir(...)` turns a docs tree into a complete, beautiful, self-contained static site — with the block fingerprints the comment layer needs.

**Depends on:** M0. **Parallel-safe with:** M2.

## Public API (exact — deviations go in DECISIONS.md)

```go
package render

type Options struct {
    Theme     Theme             // required; render.Paper() for the built-in
    Overrides map[string]string // --qbl-* token overrides, may be nil
    TOC       bool              // sidebar table of contents
    Title     string            // site title for the header; default "Docs"
}

func New(opts Options) (*Renderer, error)

// RenderDir renders every .md file in fsys (paths = slugs per conventions §Naming).
func (r *Renderer) RenderDir(fsys fs.FS) (*Site, error)

// RenderDoc renders one document. relPath is used for slug + title fallback.
func (r *Renderer) RenderDoc(src []byte, relPath string) (*Doc, error)

type Site struct {
    Docs   []*Doc
    Assets map[string][]byte // css/js served alongside; keys are site-relative paths
}
// WriteTo writes index.html, one <slug>.html per doc, and Assets into dir.
func (s *Site) WriteTo(dir string) error

type Doc struct {
    Slug    string
    RelPath string
    Title   string     // first h1, else filename
    HTML    []byte     // article body only (no chrome)
    Page    []byte     // full page: chrome + TOC + article, ready to serve
    Blocks  []Block
    Outline []Heading  // for TOC: {Level int; Text string; Anchor string}
    Text    string     // normalized plain text of the doc (anchoring input, see M2)
}

// Block is a top-level markdown block with a stable fingerprint.
type Block struct {
    ID    string // fingerprint: first 10 hex of sha256(normalized text) + "-" + occurrence index, e.g. "a3f9c210de-0"
    Text  string // normalized plain text of the block
    Start int    // rune offset of block's text within Doc.Text
    End   int
}

type Theme interface {
    Name() string
    FS() fs.FS // must contain theme.yml, tokens.css, theme.css; optional layout.html, assets/
}
func Paper() Theme
func ThemeFromFS(fsys fs.FS) (Theme, error) // validates contract; full custom themes are v0.2 but this function ships now
```

## Pipeline spec

1. **Parse** with goldmark: GFM (tables, strikethrough, task lists, autolinks) + footnotes + `WithHeadingAttribute` off — heading anchors are generated by us (below), not by author attributes.
2. **Heading anchors:** GitHub-style slugging (lowercase, alnum + hyphens, dedupe with `-1`, `-2`). Each heading gets `id` + a hover `#` anchor link.
3. **Syntax highlighting:** chroma server-side, one light + one dark stylesheet keyed off `[data-qbl-scheme]`, classes not inline styles. Unknown languages fall back to plain `<pre>` without error.
4. **Normalized text extraction** (feeds anchoring, M2): walk the AST; concatenate text content of each top-level block; within a block collapse runs of whitespace to single spaces and trim; join blocks with `"\n"`. This normalization function lives in `pkg/render` and is **exported** (`render.NormalizeBlocks`) because `pkg/comment` tests must use the identical function. Its behavior is frozen by golden tests — changing it orphans users' anchors.
5. **Fingerprints:** per top-level block, `sha256(normalizedBlockText)` → first 10 hex chars; occurrence index disambiguates identical blocks (`-0`, `-1`, …) counted in document order. Emitted into HTML as `data-qbl="<id>"` on the block's element.
6. **Chrome (layout):** header (site title, doc title, light/dark toggle), optional TOC sidebar from `Outline`, article, footer ("rendered by quibble"). Layout template is Go `html/template` from the theme's `layout.html` or the built-in default.
7. **Comment layer hooks (M5 consumes):** the layout defines empty mount points — `<div id="qbl-comments-root">` and a `<script type="module" src="qbl/comments.js">` include that 404s harmlessly under plain `build` output. `build` output is fully static and works offline with no comment UI.

## Theme contract (DESIGN.md §5.1, precise form)

- `theme.yml`: `{name, author, schemes: [light, dark]}`.
- `tokens.css` MUST define, under `:root` (light) and `[data-qbl-scheme="dark"]`, every token in this list — validation fails loudly on any missing one:
  `--qbl-bg --qbl-bg-raised --qbl-fg --qbl-fg-muted --qbl-accent --qbl-border --qbl-prose-max --qbl-font-body --qbl-font-heading --qbl-font-mono --qbl-radius --qbl-mark-bg --qbl-mark-bg-active --qbl-comment-bg --qbl-comment-border`
- `Options.Overrides` values are injected as a final `<style>:root{...}</style>` after tokens.css — override keys must be in the token list, else `New` errors.
- Scheme toggle: JS-free default via `prefers-color-scheme`; a small inline script (a few lines, part of layout) persists a manual toggle to `localStorage` and stamps `data-qbl-scheme`.

### Paper theme (the v0.1 built-in — invest real care here)

Editorial, quiet, long-form. Serif headings (Charter/Georgia stack), humanist sans body option — no; body serif too (Charter first), mono = ui-monospace stack. ~68ch prose measure, 1.65 line height, restrained accent (deep blue `#1a4fd6` light / `#7da2ff` dark), generous vertical rhythm, understated borders. Tables get horizontal-scroll wrappers. This theme is the product's face; it should look like a well-set book page, not a bootstrap page.
